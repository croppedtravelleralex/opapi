# AI.md

`api` 项目 AI / 人类接手入口文档。

目的：让任何人或 AI 在最短时间内理解这个目录是什么、当前处于什么阶段、应该先看什么、如何继续推进。

---

## 1. 项目是什么

这是一个围绕 **Sub2API / ChatGPT Business 账号池反代服务自动化** 的项目孵化目录。

当前已有内容不是代码，而是两份核心方案文档：

- `Sub2API自动化架构设计_0330224011.md`
- `UbuntuDocker部署手册_0331165925.md`

一句话：

> 这是一个从“方案/部署文档”向“真实 API 工程项目”演进的起步目录，不是已经完成的代码仓库。

---

## 2. 当前阶段

当前处于：

> **方案整理完成，准备进入工程化落地阶段。**

当前已知现状：

- 已有一份自动化架构设计文档
- 已有一份 Ubuntu + Docker 部署手册
- 目录下还没有正式代码骨架
- 当前还不是 Git 仓库
- 当前还缺统一 README、计划、状态、待办、进展等项目主文档

接手时不要把这个目录误判为“已有完整 API 服务代码”。

## 4. 接手顺序（默认）

以后任何人或 AI 接手本项目，默认按这个顺序：

1. 先看 `AI.md`
2. 再看 `PLAN.md`
3. 再看 `FEATURES.md`
4. 再看 `STATUS.md` / `PROGRESS.md`
5. 最后再回看原始方案文档

不要一上来就在历史方案里乱翻，否则很容易把“设计目标”误判成“已经落地的能力”。

---

## 5. 接手时必须确认的事

- 当前目录是否已经开始代码落地
- 当前是否已经初始化 Git 仓库
- 当前主任务是否只有一个
- 文档描述是否仍和真实阶段一致
- 是否需要补代码骨架、部署骨架或验证脚本

---

## 6. 接手原则

- 先校准文档，再推进代码
- 先确认当前真实阶段，再决定下一步
- 不把方案设计误当成已实现能力
- 不同时开太多支线
- 每次推进后同步检查文档是否需要更新
