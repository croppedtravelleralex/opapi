# RUNTIME_AND_PRODUCT_SHAPE_REVIEW.md

## 结论
当前机器与项目形态更适合做：
- **后端服务 + CLI + 轻量 dashboard + 后台 worker**

不适合做：
- 本地 GUI 重客户端
- 浏览器内复杂前端大系统
- 额外高占用图形界面

---

## 本机运行环境检查

### 系统
- Linux x86_64
- CPU: 2 vCPU
- 内存: 3.6 GiB
- Swap: 9.9 GiB
- 根盘当前使用率较高，但用户已明确后续可扩容到 `180G`

### 开发/运行工具
- Rust / Cargo 可用
- `tmux` 可用
- `jq` 可用
- `python3` 可用
- `sqlite3` 当前未发现 CLI，但 `rusqlite` 已可工作

### 当前项目形态
- 纯 Rust 服务端
- `axum + tokio + rusqlite`
- 已有 HTTP 服务、SQLite、测试
- 没有 GUI 依赖

---

## 应该砍掉的方向

### 1. GUI 桌面端
原因：
- 当前机器资源有限
- 不是主线核心价值
- 会拖慢额度反代主线
- 后续即使要看状态，dashboard 就够用

### 2. 花哨第三方 provider 市场/接入面
原因：
- 用户已明确不是当前核心
- 会分散研发资源

### 3. 复杂前端运营平台（早期）
原因：
- 早期先要系统能跑，不是先做漂亮后台

---

## 应该增加的必要功能

### 1. CLI 互动入口（强烈建议优先）
建议功能：
- 导入母号
- 导入子号
- 查看空间状态
- 触发邀请
- 触发子号验证
- 查看额度快照
- 查看池状态
- 生成/禁用 API Key
- 人工出池/恢复

原因：
- 最适合当前服务器环境
- 最快形成可操作系统
- 最容易与自动化 worker 结合

### 2. 轻量 dashboard（建议第二优先）
建议只做：
- 看板
- 搜索
- 状态查看
- 轻干预按钮

不建议早期做：
- 富编辑器
- 拖拽编排
- 重图表前端

### 3. 后台 worker / scheduler
建议功能：
- 邀请任务执行
- 子号验证任务执行
- 额度采集任务执行
- 入池/出池任务执行
- 定时健康检查

### 4. 审计查询接口
建议功能：
- 查最近出池原因
- 查某个子号最近状态变化
- 查某个外部 API Key 用了谁的额度

---

## 建议的最终产品形态

### 核心运行面
- Rust API Server
- SQLite 控制平面
- Fingerprint Browser Adapter
- Worker / Scheduler

### 人机交互面
- CLI（主要）
- Dashboard（辅助）

### 明确不做
- GUI 桌面端
- 重前端运营平台
- provider 花活优先级

---

## 推荐落地顺序
1. 先做数据模型
2. 先做 CLI
3. 再做 worker/scheduler
4. 再做 dashboard
5. 最后才考虑更复杂交互面
