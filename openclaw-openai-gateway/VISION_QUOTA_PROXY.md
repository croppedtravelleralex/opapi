# VISION_QUOTA_PROXY.md

## 结论
`openclaw-openai-gateway` 的核心目标不是继续扩“第三方 provider 接入”，而是先把 **真实额度反代** 做成，再把这些额度组织成 **安全、稳定、可治理的额度池**，最后再对外提供 **独立 API Key**。

一句话：

**先解决“怎么稳定反代出额度”，再解决“怎么把额度组织成池并安全对外提供 API Key”。**

---

## 核心主线

### 第一阶段：真实额度来源
优先额度来源：
1. `Codex App`
2. `Web` 网页端

明确暂缓：
- 第三方 API Key + BaseURL provider 扩展

### 第二阶段：额度池
把通过验证的子号组织成池：
- 入池
- 出池
- 冷却
- 恢复
- 风险隔离

### 第三阶段：对外能力
- 按用户分配独立 Key
- 统一 BaseURL
- 统一请求入口
- 审计与配额回写

---

## 业务核心判断

### 母号不是额度池成员
母号负责：
- 空间管理
- 邀请子号
- 恢复性操作

母号默认不负责：
- 承接反代流量
- 进入额度池

### 子号才是额度池成员
只有通过以下流程的子号才可入池：
1. 被母号邀请进入空间
2. 子号登录成功
3. 验证已进入目标空间
4. 网页额度读取正常
5. 风险状态正常

---

## 额度观测目标
必须同时观测：
- 5h 额度条
- 7d 额度条
- request 次数
- token
- message

说明：
- 5h / 7d 是池级健康指标
- request / token / message 是使用量、风控与调度参考信号

## 目标产品形态
这是一个：

**额度池驱动的 OpenAI 兼容代理平台**

而不是：
- GUI 桌面软件
- 通用 provider 商店
- 花哨但无核心额度来源的 demo 网关
