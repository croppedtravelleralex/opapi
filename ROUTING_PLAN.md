# ROUTING_PLAN.md

## 多上游路由演进边界（V1 -> V2）

当前阶段先只支持 **单上游透传**，目标不是一步到位做完整调度系统，而是先把网关主链打通。

---

## V1：单上游模式

能力：
- 一个 `UPSTREAM_BASE_URL`
- 一个 `UPSTREAM_API_KEY`
- 所有 `/v1/chat/completions` 请求统一转发到同一个上游

用途：
- 验证网关最小闭环
- 验证鉴权、错误映射、联调链路
- 验证 OpenAI 兼容接口形状

---

## V1.5：静态多上游配置

能力：
- 多个 provider 配置
- 按模型静态映射到指定上游
- 不做动态调度，只做确定性路由

示例：
- `gpt-5.4 -> date-now`
- `qwen3-max -> iflow`
- `gpt-4.1-mini -> asxs`

用途：
- 先解决“一个网关挂多个真实上游”
- 为后续动态调度做结构预留

---

## V2：动态路由模式

能力：
- 同模型多上游候选
- 按健康状态、错误率、优先级、手动权重动态选择
- 支持 fallback

用途：
- 提高可用性
- 为账号池 / 通道池做基础调度层

---

## V3：账号池与健康系统

能力：
- 账号健康检查
- 账号/通道配额感知
- 冷却、禁用、恢复
- 更细粒度审计

---

## 当前原则

1. 先把 V1 做实，不虚胖
2. 再做 V1.5 静态多上游
3. 再推进 V2 动态路由
4. 账号池能力放在 V3，不提前压进当前阶段
