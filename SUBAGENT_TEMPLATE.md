# SUBAGENT_TEMPLATE.md

子 agent 标准调用模板（默认规范）

> 目标：让子 agent **更稳定、结果更可用、回收更可靠**。
> 默认适用于：排查、审查、方案设计、独立分析、小型调研。
> 默认**不适用**：需要持续多轮互动的持久会话、强依赖 thread/session 绑定的任务。

---

## 0. 默认策略

### 一律默认
- `runtime: "subagent"`
- `mode: "run"`
- `sandbox: "inherit"`
- `cleanup: "delete"`
- `thread: false`
- `thinking: "medium"`
- `model: "default"`

### 不默认使用
- `mode: "session"`
- `thread: true`
- ACP runtime

### 原因
当前环境下，**一次性 run 模式最稳**；`session` 模式和 thread 绑定在部分目标下不稳定，不作为默认路径。

---

## 1. 任务选择规则

适合开子 agent：
1. **独立排查**：某个问题可以单独分析，不需要主线程持续来回互动
2. **代码/脚本审查**：读取若干文件、找 bug、找漂移、提建议
3. **方案设计**：让它给出实现顺序、文件级改动建议、验收标准
4. **信息归纳**：把多个来源整理成短结论

不适合开子 agent：
1. **超短任务**：主线程 1-2 个工具调用就能做完
2. **强依赖上下文手感**：需要连续理解用户即时反馈
3. **必须持久 thread/session** 的任务
4. **对输出格式极其敏感但又没有明确约束** 的任务

---

## 2. 标准调用参数模板

```json
{
  "agentId": "main",
  "attachAs": { "mountPath": "/workspace" },
  "attachments": [],
  "cleanup": "delete",
  "cwd": "/absolute/project/path",
  "label": "short-clear-task-name",
  "mode": "run",
  "model": "default",
  "resumeSessionId": "",
  "runTimeoutSeconds": 900,
  "runtime": "subagent",
  "sandbox": "inherit",
  "task": "<structured task prompt>",
  "thinking": "medium",
  "thread": false,
  "timeoutSeconds": 900
}
```

### 参数备注
- `label`：短、清楚、可识别，例如：
  - `rust-env-check`
  - `verify-script-review`
  - `integration-plan`
- `cwd`：直接指向项目根目录，不要含糊
- `runTimeoutSeconds` / `timeoutSeconds`：默认 `900`，小任务可降到 `300`

## 3. 标准任务提示词模板

所有子 agent 任务，默认都按这个结构写。

```text
你是一个子 agent，负责处理单一明确任务。

任务目标：
<一句话说明要它完成什么>

工作范围：
- 只处理：<范围>
- 不要处理：<边界>
- 不要做破坏性改动
- 不要安装软件
- 如需改文件，只改与目标直接相关的最小范围

上下文：
- 项目路径：<path>
- 你应重点查看：<文件/目录>
- 当前已知约束：<约束>

输出要求：
必须按下面格式输出，不能留空：
1. 结论：<一句话结论>
2. 发现：
   - <发现1>
   - <发现2>
3. 建议：
   - <建议1>
   - <建议2>
4. 若你做了修改：
   - 修改了哪些文件
   - 修改目的是什么
5. 若你未能完成：
   - 卡在哪
   - 还缺什么
```

## 4. 三类高频模板

### A. 排查型模板

```text
你是一个排查子 agent。

任务目标：定位 <问题> 的真实原因，并给出最短修复建议。

工作范围：
- 只排查 <范围>
- 不要安装软件
- 不要做破坏性改动
- 允许读取文件、执行低风险检查命令

上下文：
- 项目路径：<path>
- 重点查看：<files>
- 当前症状：<symptom>

输出要求：
1. 结论：<最可能根因>
2. 发现：
   - <证据1>
   - <证据2>
3. 建议：
   - <最短修复路径>
   - <备选路径>
4. 风险：
   - <改动风险>
5. 若未完成：
   - <阻塞点>
```

### B. 审查型模板

```text
你是一个审查子 agent。

任务目标：审查 <脚本/模块> 是否自洽，找出明显 bug、漂移和最值得先修的问题。

工作范围：
- 只审查：<scope>
- 不要安装软件
- 不要做大改
- 可做静态检查和低风险验证

输出要求：
1. 结论：<整体质量判断>
2. 发现：
   - <问题1>
   - <问题2>
   - <问题3>
3. 建议：
   - <最优先修复项>
   - <次优先修复项>
4. 若做了验证：
   - <验证方式>
   - <验证结果>
```

### C. 方案设计型模板

```text
你是一个方案设计子 agent。

任务目标：为 <目标> 给出最小落地方案，按优先级排序。

工作范围：
- 只做方案设计
- 先不要改代码
- 方案要尽量具体到文件、数据结构、接口行为、验收标准

输出要求：
1. 结论：<推荐方案>
2. 方案：
   - 第1步：<...>
   - 第2步：<...>
   - 第3步：<...>
3. 文件建议：
   - <文件1>: <改什么>
   - <文件2>: <改什么>
4. 验收标准：
   - <标准1>
   - <标准2>
5. 风险与取舍：
   - <风险>
```

## 5. 结果回收规则

子 agent 完成后，主线程默认这样处理：

1. **先看有没有结构化输出**
   - 有：直接收口给用户
   - 没有：不要假装它完成了有价值工作

2. **如果返回空结果**
   - 视为“任务机制跑通，但产出无效”
   - 必须明确告诉用户：
     - 子 agent 已运行
     - 但没有返回有效结论

3. **必要时做二次兜底**
   - 可以查会话历史
   - 或主线程自行补审
   - 但不要把空结果包装成成功产出

---

## 6. 默认质量门槛

以后开子 agent，至少满足：
1. **任务目标单一**
2. **输出格式明确**
3. **边界明确**
4. **不要让它自由散写**
5. **不要默认依赖 session 模式**
6. **结果为空时必须诚实回报**

---

## 7. 推荐默认话术（给自己用）

### 启动前内部判定
- 这个任务是否值得开子 agent？
- 是否可以拆成单一明确目标？
- 是否已经给出明确输出格式？

### 对用户的简短说明
- “我开一个子 agent 专门做 `<任务>`，你主线程这边我来收口。”
- “我会让它按固定格式回结论，避免空跑。”

---

## 8. 当前环境特殊注意事项

1. 当前环境里：
   - **`runtime=subagent + mode=run` 是默认稳定路径**
   - **`mode=session` 不作为默认方案**
2. 若结果回传为空，不代表没跑，而是**结果质量不合格**
3. 若未来 ACP backend 配好，可再单独定义 ACP 模板；不要混进这个默认模板
